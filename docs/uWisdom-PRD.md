## 一、项目概述

### 1.1 项目背景

当前个人知识管理普遍存在以下痛点：

- 知识零散化，存储在多平台多格式中，无法形成体系化、可迭代的知识资产
- 知识仅面向人类可读，缺乏机器可解析的结构化设计，无法被 AI Agent 直接调用
- 知识演进无版本追溯，无法验证知识的有效性与适用边界
- 知识与商业落地脱节，难以快速复用支撑实际业务场景

基于此，uWisdom 项目以**驾驭工程可读性原则**为核心，构建一套人机协同的个人专属领域知识百科系统，实现知识的资产化、可复用、可执行。

### 1.2 核心定位

uWisdom 是**基于驾驭工程可读性原则的个人知识操作系统**，以 GitHub 为唯一托管载体，以 GitHub Pages 为发布出口，打造“人类可读可维护、机器可读可调用、版本可追溯、场景可落地”的专属领域知识百科。

### 1.3 愿景与目标

#### 长期愿景

成为个人知识资产化的标准框架，实现知识从沉淀、演进、检索到商业落地的全生命周期管理，打造人机协同的知识生产力底座。

#### 核心业务目标

1. 实现个人知识的**可持续演进**：基于 Git 的版本化管理，完整追溯知识的迭代全流程
2. 实现知识的**智能检索**：支持结构化、关联化、语义化的知识查询，快速定位目标内容
3. 实现**Agent 可读可调用**：标准化的知识结构，支持 AI Agent 直接读取、解析、引用知识
4. 实现商业场景**可控落地**：模块化的知识设计，可快速复用支撑商业项目，保障知识的可验证、可追溯

`Area（领域） -> Project（项目） -> Task（任务）`

- `Area` 对应长期稳定的关注方向
- `Project` 对应阶段性建设主题

uWisdom 不仅要存储静态知识，还要承接从认知到行动的连续链路。

## 三、核心设计原则

项目全流程严格遵循**驾驭工程可读性原则**，将知识视为“可执行的代码”，落地以下 5 个核心原则：

1. **模块化**：知识拆分为独立、低耦合、可复用的“知识单元”，类似代码中的函数或类
2. **结构化元数据**：每个知识单元附带机器可读的标准化元数据，类似代码的注释与类型定义
3. **版本可追溯**：基于 Git 完整记录知识的创建、修改、迭代历史，类似代码的提交日志
4. **依赖关系显式化**：知识间的关联通过链接、引用显式表达，类似代码的 import 依赖
5. **测试用例化**：每个知识单元附带验证示例，类似代码的单元测试，保障知识的有效性与适用边界

在此基础上补充两条产品级原则：

6. **人机双读**：既适合人阅读，也适合 Agent 解析、召回与调用
7. **机会导向**：围绕长期价值发现组织知识，而不是围绕短期问题堆积信息

## 四、整体架构设计

### 4.1 仓库结构设计

项目采用单仓库架构，全量内容托管于 GitHub，整体结构如下：

```text
uWisdom/
├── .github/                    # GitHub 配置目录
│   ├── workflows/              # CI/CD 自动化工作流
│   ├── ISSUE_TEMPLATE/         # 知识需求/问题提交模板
│   └── PULL_REQUEST_TEMPLATE/  # PR 审核模板
├── docs/                       # GitHub Pages / Docusaurus 站点源文件
│   ├── index.md                # 站点首页
│   ├── uWisdom-PRD.md          # 产品需求文档
│   ├── knowledge-base/         # 核心知识库
│   │   ├── areas/              # Area 层：长期领域
│   │   ├── projects/           # Project 层：阶段主题
│   │   ├── tasks/              # Task 层：执行任务模板与实践记录
│   │   ├── principles/         # 原则库
│   │   ├── methods/            # 方法库
│   │   ├── glossary/           # 术语库
│   │   ├── decisions/          # 决策库
│   │   └── cases/              # 案例库
│   ├── identities/             # 核心身份地图
│   ├── skills/                 # Agent 可调用技能库（关联 uSkills 项目）
│   ├── schemas/                # 知识结构标准 Schema
│   └── assets/                 # 图片、附件等静态资源
├── tools/                      # 项目维护工具脚本
│   ├── knowledge-generator/    # 知识单元模板生成工具
│   ├── agent-sync/             # Agent 接口同步工具
│   └── validator/              # 知识结构合规性验证工具
├── examples/                   # 商业场景落地案例集
├── README.md                   # 项目说明文档
└── LICENSE                     # 开源协议（默认 MIT）
```

### 4.2 技术栈选型

| 模块 | 技术选型 | 选型说明 |
|------|----------|----------|
| 静态站点生成器（SSG） | Docusaurus | 前端工程化能力强，原生支持文档版本化、插件化扩展，可补全双向链接、索引、知识图谱能力，适配长期演进 |
| 智能检索 | Algolia DocSearch | 成熟的文档检索方案，支持全文检索，与 Docusaurus 原生兼容 |
| 知识图谱可视化 | D3.js + Mermaid | 支持自定义知识关联图谱渲染，可嵌入 Markdown 文档 |
| Agent 兼容层 | JSON Schema + GitHub API | 定义标准化知识结构，通过 GitHub API 开放给 Agent 读取与调用 |
| 自动化 CI/CD | GitHub Actions | 实现知识验证、站点构建、Pages 发布、Agent 接口同步自动化 |
| 版本管理 | Git + GitHub Releases | 实现知识版本化管理，重要迭代节点发布 Release 版本 |

### 4.3 核心数据流

1. **知识生产流**：提交知识需求 -> 生成知识单元模板 -> 编写内容 -> 合规性验证 -> PR 审核 -> 合并 `main`
2. **站点发布流**：`main` 更新 -> 触发 GitHub Actions -> 验证知识结构 -> 构建 Docusaurus 站点 -> 发布到 GitHub Pages
3. **Agent 调用流**：Agent 通过 GitHub API 获取知识 Schema -> 检索目标知识单元 -> 解析元数据与内容 -> 关联调用对应技能 -> 输出结果并记录调用日志

## 五、领域信息架构设计

### 5.1 顶层知识分层

uWisdom 的知识组织需要同时支持“稳定认知”和“动态执行”，因此采用五层结构：

1. 身份层：我是谁，我长期扮演哪些角色
2. 领域层：我长期关注哪些 Area
3. 方法层：我如何思考、判断、行动
4. 项目层：我当前推进哪些主题 Project
5. 任务层：我每周、每天实际执行哪些 Task

### 5.2 身份层设计

核心身份建议拆为四个一级入口：

- 架构师
- 投资人
- 终身学习者
- 生活艺术家

每个身份页应包含：

- 身份定义
- 长期目标
- 关键能力
- 关注领域
- 当前项目
- 代表性知识链接

### 5.3 Area 层设计

Area 是知识组织的第一稳定锚点，建议使用以下规则：

- 每个 Area 表达一个长期持续的价值方向
- 一个 Area 下可以有多个 Project
- Area 页面重点呈现目标、边界、指标、当前状态与相关知识

示例 Area：

- 系统架构设计
- AI 原生与云原生转型
- 投资组合管理
- 另类投资探索
- 持续技能提升
- 生活品质提升

### 5.4 Project 层设计

Project 是 Area 的阶段性实现单元。每个 Project 文档建议包含：

- 项目目标
- 背景与机会
- 关键假设
- 相关知识单元
- 当前任务列表
- 阶段性成果

### 5.5 Task 层设计

Task 不只是待办项，还应成为知识反馈回路的一部分。

每个 Task 应记录：

- 所属 Area / Project
- 任务动作
- 执行日期
- 执行结果
- 新增洞见
- 是否沉淀为知识单元

### 5.6 机会导向索引

由于你的驱动力是“挖掘机会”，uWisdom 需要新增一类机会索引页：

- 趋势机会
- 技术机会
- 投资机会
- 生活方式机会
- 跨领域机会

该索引用于把“发现机会”从零散灵感提升为可跟踪、可评估、可转化的知识流。

## 六、详细功能设计

### 6.1 知识库管理模块（P0）

#### 6.1.1 知识单元标准化管理

- 功能描述：定义统一的知识单元规范，采用“Markdown + Front Matter”格式存储，每个知识单元包含唯一 ID、元数据、正文内容、验证示例、场景引用
- 核心要求：
  - 元数据必须包含 `id`、`title`、`field`、`tags`、`version`、创建/更新时间、依赖关系、Agent 可调用标识、关联技能引用
  - 正文必须包含定义、核心规则、验证示例、应用场景四个基础模块
  - 支持双向链接语法，可引用其他知识单元
- 验收标准：所有知识单元符合统一规范，可被工具自动验证、解析

#### 6.1.2 知识版本化管理

- 功能描述：基于 Docusaurus 原生版本能力和 Git 提交记录，实现知识的全版本追溯
- 核心要求：
  - 支持知识的大版本归档，如 `v1.0`、`v2.0`
  - 基于 Git 提交记录，可查看每个知识单元的修改历史、修改人、修改内容
  - 支持版本回滚，可恢复历史版本内容
- 验收标准：可在站点访问历史版本知识库，可追溯单个知识单元的完整迭代历史

#### 6.1.3 知识分类与标签管理

- 功能描述：支持按领域、身份、标签对知识单元进行分类管理，实现多维度知识导航
- 核心要求：
  - 一级分类同时支持“身份分类”和“领域分类”
  - 二级分类支持子领域、专题与方法维度
  - 支持自定义标签，可通过标签筛选关联的知识单元
- 验收标准：站点可通过身份、领域、标签快速筛选目标知识单元

### 6.2 站点与可视化模块（P0）

#### 6.2.1 GitHub Pages 站点发布

- 功能描述：基于 Docusaurus 构建静态站点，自动发布到 GitHub Pages
- 核心要求：
  - 站点包含首页、知识库、技能库、商业案例、关于项目五个核心页面
  - 首页展示知识图谱、热门知识、最新更新内容与机会索引
  - 响应式设计，适配 PC 和移动端访问
- 验收标准：`main` 合并后自动完成站点构建与发布，可通过 GitHub Pages 域名正常访问

#### 6.2.2 知识图谱可视化

- 功能描述：基于知识单元的依赖关系，生成可视化的知识关联图谱
- 核心要求：
  - 支持全局知识图谱和单知识单元关联图谱
  - 点击图谱节点可跳转到对应知识单元页面
  - 支持缩放、拖拽等交互操作
- 验收标准：图谱可正确展示知识间的关联关系，交互正常

#### 6.2.3 双向链接支持

- 功能描述：支持 Wiki 双向链接语法，自动解析知识单元间的引用关系
- 核心要求：
  - 支持 `[[知识单元ID]]` 格式的双向链接
  - 知识单元详情页展示“被哪些知识单元引用”的反向链接列表
- 验收标准：双向链接可正常跳转，反向链接列表可正确展示

### 6.3 智能检索模块（P1）

#### 6.3.1 全文检索

- 功能描述：接入 Algolia DocSearch，实现知识库的全文检索
- 核心要求：
  - 支持标题、正文、标签、领域的多维度检索
  - 检索结果按相关性排序，高亮匹配关键词
- 验收标准：可通过检索框快速定位目标知识单元，检索响应时间小于 1 秒

#### 6.3.2 语义化检索（远期 P2）

- 功能描述：支持基于语义的知识检索，匹配含义相关的知识单元
- 核心要求：
  - 支持自然语言提问，返回相关知识单元
  - 可识别知识间关联关系，推荐相关内容
- 验收标准：可理解自然语言查询，返回语义匹配的知识内容

### 6.4 Agent 兼容模块（P1）

#### 6.4.1 标准化知识 Schema

- 功能描述：基于 JSON Schema 定义知识单元的标准结构，提供机器可读的规范
- 核心要求：
  - Schema 定义元数据的必填字段与格式规范
  - 支持版本迭代，兼容历史版本知识单元
- 验收标准：所有知识单元符合 Schema 规范，可被机器自动解析

#### 6.4.2 Agent 调用接口

- 功能描述：基于 GitHub API 开放知识读取接口，支持 Agent 直接调用
- 核心要求：
  - 支持按 ID、领域、标签检索知识单元
  - 支持获取知识单元的完整元数据与正文内容
  - 支持获取知识单元的关联依赖与引用关系
- 验收标准：Agent 可通过 GitHub API 正常获取、解析知识内容

### 6.5 商业场景支撑模块（P1）

#### 6.5.1 商业案例库

- 功能描述：提供知识落地的商业案例，包含完整的知识引用、实现流程、效果说明
- 核心要求：
  - 每个案例关联对应的知识单元与技能
  - 支持一键复用案例到实际项目
- 验收标准：案例可完整复现，可清晰追溯引用的知识内容

#### 6.5.2 权限与可控复用

- 功能描述：基于 GitHub 权限体系，实现知识的可控复用
- 核心要求：
  - 支持公开/私有仓库切换，控制知识访问权限
  - 支持通过 Git Submodule 将知识模块引入商业项目
  - 可追溯知识在商业项目中的引用情况
- 验收标准：可灵活控制访问权限，可通过 Submodule 实现知识复用

### 6.6 自动化运维模块（P0）

#### 6.6.1 知识合规性自动验证

- 功能描述：基于 validator 工具，自动验证 PR 中知识单元的格式合规性
- 核心要求：
  - 验证元数据必填字段是否完整、格式是否正确
  - 验证双向链接是否有效，无死链
  - 验证不通过的 PR 自动拦截，并给出修改提示
- 验收标准：不符合规范的知识单元无法合并到 `main`

#### 6.6.2 自动化发布流程

- 功能描述：基于 GitHub Actions 实现全流程自动化发布
- 核心要求：
  - `main` 合并后自动触发构建、验证、发布流程
  - 发布完成后自动同步知识元数据到 Agent 接口
  - 构建失败自动发送通知
- 验收标准：全流程无需人工干预，发布完成后站点内容自动更新

## 八、与 uSkills 项目的协同设计

uWisdom 与 uSkills 为互补协同项目，明确分工如下：

- **uWisdom**：知识底座，存储“是什么 / 为什么 / 用在哪”的知识内容，是认知层
- **uSkills**：技能库，存储“怎么做 / 如何执行”的可执行技能，是执行层

协同规则：

1. uWisdom 的知识单元通过 `skill_ref` 字段关联 uSkills 中的对应技能
2. uSkills 的技能通过 `knowledge_ref` 字段引用 uWisdom 中的对应知识
3. Agent 可链式调用：先从 uWisdom 获取知识，理解规则与边界，再调用 uSkills 完成执行
4. 两个项目通过 Git Submodule 或版本引用实现关联，保持版本同步

## 十、风险评估与应对方案

| 风险类型 | 风险描述 | 应对方案 |
|----------|----------|----------|
| 结构风险 | 知识单元规范设计不合理，导致后续扩展困难 | 前期充分验证 Schema 的兼容性，预留扩展字段，大版本迭代前充分测试 |
| 兼容风险 | Agent 调用接口适配性不足，无法被主流 Agent 框架兼容 | 遵循业界通用的 OpenAPI / JSON Schema 思路，提供主流 Agent 框架接入示例 |
| 性能风险 | 知识量增长后，站点构建速度、检索速度变慢 | 优化构建流程，实现增量构建；优化检索引擎，支持分片检索 |
| 维护风险 | 自动化流程故障，导致站点发布失败 | 完善告警机制，保留手动发布兜底方案，定期测试工作流可用性 |
| 范围风险 | 个人兴趣面过广，导致体系扩张失控 | 以 Area 为边界，按优先级建设，优先沉淀高复用、高价值领域 |

## 十二、结论

uWisdom 的本质不是“知识展示网站”，而是“以 GitHub 为底座、以工程可读性为核心、以 Area -> Project -> Task 为骨架、以 Agent 可调用为扩展方向的个人知识操作系统”。

它以你的真实领域结构为第一性输入，把职业演进、兴趣成长、投资判断、生活实践统一收束到一个持续演进的知识工程中，最终形成可读、可检索、可调用、可发布、可商业化复用的长期资产。
